As I've grown as a developer I've found that a lot of the hard work I have to do is to keep my own cleverness in check. As a senior developer and now especially as a manager, I am learning to favor readability and maintainability of code over succinct and dry code.

Thus, one thing I find myself suggesting a lot in code reviews these days is to ditch an abstraction around the method or API of a dependency. Let me give an example of what I mean.

- don't always abstract
- building up juniors
- developers should learn the API / library / framework vs some tribal code that doesn't transfer professionally or to other projects
- can you exist without your abstractions?
- libraries etc are documented and maintained, your custom code is not
